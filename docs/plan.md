# 航空機写真AI選別ソフト 要件定義書兼基本設計書

## 1. 文書情報

* 文書名: 航空機写真AI選別ソフト 要件定義書兼基本設計書
* アプリケーション名: SkySort
* リポジトリ名: skysort
* 版数: v1.3
* 想定読者: 発注者、実装担当者、レビュー担当者
* 目的: ローカルPC上で動作し、大量の航空機写真をAIと画像処理で選別・評価・確認できるソフトウェアの実装を委託するための要件および基本設計を定義する

---

## 2. 背景と目的

### 2.1 背景

発注者は、空港周辺で航空機写真を撮影する際に、ミラーレスカメラと望遠ズームレンズを用いて1日あたり数千枚規模の写真を撮影する。撮影後には、以下の作業に大きな時間を要している。

* 連写された類似写真の中から最も良い1枚を選ぶ
* 失敗カットを判定して削除候補を抽出する
* 良い写真にレーティングを付与する
* 後工程の現像・保存対象を絞り込む

### 2.2 目的

本ソフトウェアは、ローカル環境で動作するマルチモーダルLLM/VLMおよび画像処理を活用し、以下を実現することを目的とする。

* 写真フォルダを読み込み、撮影画像を一括解析する
* 類似写真をグルーピングし、連写バースト単位で比較・評価する
* 技術的品質と見栄えの両面から写真にスコアを付与する
* 最終的に各画像へ★1〜★5または reject を付与する
* ユーザーが結果を確認・修正できるUIを提供する
* 必要に応じてXMP Rating等へ結果を書き戻す

### 2.3 本プロジェクトの基本方針

* 完全ローカル動作を前提とする
* AI推論サーバーはLM StudioのOpenAI互換APIを第一候補とする
* ただし、将来vLLM等へ差し替え可能な疎結合設計とする
* AI単独判断に依存せず、画像処理による定量指標と組み合わせる
* 写真全体の絶対評価より、連写グループ内の相対比較を重視する
* 最終判断はユーザーが確認・修正できる前提とする

### 2.4 用語定義

* グループ: 時刻近接性および画像類似度に基づいてまとめられた写真集合
* バースト: 連写により短時間で撮影された写真群。本書ではグループの代表的な発生単位として扱う
* ベストカット: グループ内で最も残す価値が高いと判定された1枚
* Pick: 利用者が手動で「残す候補」として強調指定した状態。★評価とは別の補助フラグ
* reject: 現像・保存候補から外すべき写真を示す状態。★1とは別概念として扱う
* 解析ジョブ: 対象フォルダ、設定スナップショット、結果を束ねる解析実行単位。同一 `root_path` でも設定変更を伴う全件再解析は別ジョブとして扱う
* 技術評価: ブレ、露出、白飛び、黒つぶれ等の定量指標に基づく評価
* 意味評価: 構図、見栄え、被写体の収まり、グループ内差分等をAIが判断する評価

## 3. 対象範囲

### 3.1 対象業務

* ローカルフォルダ内の写真の取り込み
* サムネイル生成
* メタデータ抽出
* 類似写真グルーピング
* 画質評価
* AIによる意味的評価
* レーティング付与
* 結果表示および手動修正
* メタデータ書き戻し

### 3.2 対象ファイル形式

初版で対象とする入力ファイル形式は以下とする。

* JPEG（.jpg, .jpeg）
* PNG
* Sony RAW（.arw）

RAWについては、推論時には内部でプレビューJPEGへ変換して利用する。初版で必須対応とするRAW形式はARWのみとし、ARW内蔵JPEGプレビューの抽出を第一選択、抽出不可または破損時のみ rawpy/libraw 等によるデモザイクへフォールバックする。将来的に他RAW形式へ拡張しやすい構造を前提とする。

### 3.3 対象外

初版では以下を対象外とする。

* クラウド同期
* 複数ユーザー同時利用
* モバイル対応
* Lightroomプラグイン化
* 学習済み独自モデルの再訓練
* 動画ファイルの解析
* クラウドAPIへの画像送信を伴う処理
* Adobe製品との高度な双方向連携
* PNGファイルへのXMP書き戻し

## 4. システム概要

### 4.1 システム名称

正式名称: SkySort

補足:
本書ではアプリケーション名を `SkySort`、リポジトリ名を `skysort` として扱う。

### 4.2 システム形態

* ユーザーPC上で起動するローカルWebアプリケーション
* バックエンドはlocalhostでAPIを提供する
* フロントエンドはlocalhostでブラウザ表示する
* 開発時は FastAPI と Vite dev server を別プロセスで起動する
* AIモデルは別プロセスで稼働するLM Studio等を利用する

### 4.3 想定利用環境

* OS: Windows 11を主対象、必要に応じてmacOS対応を考慮
* GPU: NVIDIA RTX 5090相当
* AIモデル: ローカル動作するマルチモーダルLLM/VLM
* AIサーバー: LM StudioのOpenAI互換API

## 5. 利用者とユースケース

### 5.1 利用者

* 主利用者: 写真撮影者本人
* 管理者機能は不要
* 複数権限管理は不要

### 5.2 主要ユースケース

#### UC-01 写真フォルダを取り込む

利用者はフォルダを指定し、画像群をシステムへ登録する。

#### UC-02 解析を実行する

利用者は対象フォルダに対して解析ジョブを開始し、進捗を確認する。

#### UC-03 類似写真グループを確認する

利用者は連写グループごとに画像を閲覧し、AIが選定したベストカットを確認する。

#### UC-04 レーティングを修正する

利用者はAIが付与した★1〜★5や reject を手動で変更する。

#### UC-05 結果を書き出す

利用者は最終評価をXMP Ratingとして写真ファイルへ書き戻す。

#### UC-06 削除候補を確認する

利用者は reject や★1の写真を一覧表示し、削除候補として確認する。

## 6. 機能要件

本章は要件レベルの定義であり、記載した振る舞いを満たす限り実装方式の詳細は実装担当者の判断で調整可能とする。

### 6.1 写真取り込み機能

#### 要件ID: FR-001

* 利用者はローカルフォルダを指定できること
* 初版ではフォルダパスの直接入力を必須手段とし、OSネイティブのフォルダ選択ダイアログは任意実装とすること
* 入力された `root_path` は正規化、シンボリックリンク解決、存在確認、通常ディレクトリ確認を行った上で受け入れること
* システムは指定フォルダを再帰的に走査できること
* 対応拡張子のファイルを検出できること
* 既登録ファイルと未登録ファイルを識別できること

#### 要件ID: FR-002

* システムは画像ファイルごとに一意な内部IDを採番すること
* システムはファイルパス、ファイルサイズ、更新時刻、ハッシュ値を記録できること
* 初版の高速ハッシュは `xxh3_64(先頭1MB + file_size)` を採用し、完全一致確認が必要な場合のみ完全ハッシュへフォールバックできること
* 再取り込み時は、新規追加ファイル、更新済みファイル、削除済みファイルを識別できること
* 更新済みファイルは再解析対象として扱い、削除済みファイルは missing 状態として扱えること

### 6.2 メタデータ抽出機能

#### 要件ID: FR-003

* システムはEXIF等から以下のメタデータを抽出できること

  * 撮影日時
  * カメラ機種
  * レンズ名
  * 焦点距離
  * シャッター速度
  * 絞り
  * ISO
  * サブ秒時刻（取得可能な場合）
  * 画像サイズ
  * 向き

### 6.3 プレビュー生成機能

#### 要件ID: FR-004

* システムはRAWを含む入力画像からサムネイルを生成できること
* システムはAI推論用の中解像度JPEGプレビューを生成できること
* 既生成済みプレビューを再利用できること
* 初期値としてサムネイルは長辺512px、推論用プレビューは長辺1024px、JPEG品質90を推奨すること
* 複数画像比較時は、モデル制約に応じて推論用プレビューを長辺512pxまで自動縮小できること
* プレビュー生成時は EXIF orientation を適用した表示向きへ正規化すること
* 推論用プレビューはsRGBへ変換して生成すること

### 6.4 類似写真グルーピング機能

#### 要件ID: FR-005

* システムは撮影時刻の近接性、画像類似度、視覚特徴量を用いて類似写真をグルーピングできること
* グループは主に連写バースト単位で構成すること
* グループは1枚のみでも成立可能とすること

#### 要件ID: FR-006

* 利用者はグルーピング結果を確認できること
* 初版では手動のグループ結合・分割・除外は対象外とし、Phase 2 で対応すること

### 6.5 技術評価機能

#### 要件ID: FR-007

* システムは画像処理により各画像の技術的品質を評価できること
* 初版では少なくとも以下を評価対象とすること

  * ブレ・シャープネス
  * 露出破綻
  * 白飛び率
  * 黒つぶれ率
* シャープネスは画像全体一律ではなく、中央重点領域または高エッジ領域を優先して算出できること
* 白飛びと黒つぶれの閾値は設定可能とし、初期値はそれぞれ輝度 252/255 以上、3/255 以下を推奨すること

#### 要件ID: FR-008

* システムは各評価項目を数値スコアとして保持できること
* システムは総合技術スコアを算出できること

### 6.6 AI意味評価機能

#### 要件ID: FR-009

* システムはローカルAIサーバーへ画像とプロンプトを送信し、意味的評価を取得できること
* OpenAI互換APIでの画像送信は `image_url` の data URL 形式による base64 インライン送信を標準とすること
* 評価は画像単位だけでなく、同一グループ内の相対比較として実行できること
* AIプロンプトは外部テンプレートファイルとして管理し、テンプレート名およびバージョンを識別できること

#### 要件ID: FR-010

* AIは少なくとも以下の観点から評価できること

  * 機体がきれいに収まっているか
  * 機首・主翼・尾翼が切れていないか
  * 構図や迫力が良いか
  * 似たカットの中で残す価値が高いか
  * ブレやピンぼけが目立たないか
  * 他写真と比較して差分があるか
* 単画像評価は全件必須とせず、技術評価とグループ内代表性をもとに候補へ優先実行し、明らかな低品質画像はスキップ可能とすること
* グループ内比較は、候補絞り込み後に少数候補をまとめて比較する二段階方式で実行できること
* 1リクエストあたりの送信画像数は最大6枚とすること

#### 要件ID: FR-011

* AI評価結果はJSON形式で取得・保存できること
* JSON化に失敗した場合は、JSONモード指定、JSONブロック抽出、最大2回の再試行を行い、それでも失敗した場合は `ai_eval_failed` として記録できること
* AI応答には JSON schema version を含め、保存時に識別できること
* 少なくとも以下を含むこと

  * グループ内順位
  * 推奨ベスト画像ID
  * 各画像の意味スコア
  * 簡潔な理由文
  * モデル名
  * プロンプトテンプレート名またはハッシュ

### 6.7 総合レーティング機能

#### 要件ID: FR-012

* システムは技術評価とAI意味評価を統合して、各画像へ★1〜★5または reject を付与できること
* 評価基準は設定ファイルで調整可能とすること
* AI評価未完了時は、技術評価ベースの暫定★評価を表示し、AI完了後に総合評価へ更新できること
* 最終評価状態は `rating` と `selection_status` で表現し、`selection_status = rejected` の場合は `rating` と両立させないこと

#### 要件ID: FR-013

* 各グループのベストカットは1枚のみ選出できること
* ベストカットとは別に、必要に応じて複数枚を高評価とできること

### 6.8 UI機能

#### 要件ID: FR-014

* 利用者は解析ジョブの進捗状況を確認できること
* 状態には、総ファイル数、処理済件数、失敗件数、現在処理中工程を含むこと

#### 要件ID: FR-015

* 利用者はグループ一覧と各画像のサムネイルを閲覧できること
* 利用者はAIスコア、技術スコア、暫定評価または最終評価、理由を確認できること

#### 要件ID: FR-016

* 利用者は手動で以下を実行できること

  * ★評価変更
  * reject 指定
  * pick 指定
  * ベストカット指定
  * 再解析
* グループ修正は Phase 2 の機能として扱うこと

#### 要件ID: FR-017

* 利用者はキーボードショートカットを用いて高速に選別できること
* 初版ショートカットはグループ詳細レビュー画面および全体レビュー画面で有効とすること
* 初版ショートカット例

  * 1〜5: 星評価
  * X: reject
  * P: pick
  * 矢印キー: グループ詳細では写真移動、全体レビューでは選択中サムネイル移動
  * Space: 拡大表示

### 6.9 結果出力機能

#### 要件ID: FR-018

* システムは最終評価結果をDBへ保存できること
* システムはCSVまたはJSONでエクスポートできること

#### 要件ID: FR-019

* システムはXMP Ratingへの書き戻しを実行できること
* 書き戻し前にプレビュー確認を行えること
* 書き戻し対象を絞り込めること
* XMP書き戻しは ExifTool 実行可能環境を前提とし、未導入時は書き戻し機能のみを無効化して他機能は継続利用できること
* 星評価および reject は標準互換タグを優先し、pick、best_cut、reviewed は SkySort 独自 namespace で保持すること
* ARWファイルへの直接書き込みは行わず、サイドカーXMPのみを更新すること
* JPEGファイルは評価関連タグに限り埋め込みXMP更新を許容すること
* PNGファイルは初版のXMP書き戻し対象外とすること
* 既存XMPが存在する場合は、評価関連タグのみを保守的に更新し、他の既存タグは維持すること
* `dry_run` は既定で有効とし、JPEGへの実書き込みは差分確認または明示確認後にのみ実行すること

### 6.10 再実行・再評価機能

#### 要件ID: FR-020

* 利用者は対象画像や対象グループのみを再評価できること
* 利用者はキャッシュを利用して全件再解析を避けられること
* 入力ファイル更新、設定変更、再解析対象の選択に応じて評価状態を stale として扱い、明示的な再解析を促せること
* 写真単位再解析をAPIから明示的に起動できること
* 手動変更された `rating`、`selection_status`、`pick`、`best_cut` は、明示的にリセットしない限り自動再解析で上書きしないこと
* 設定変更を反映した全件再解析は新しい解析ジョブとして扱うこと
* Phase 2 のグループ修正後は影響範囲の評価状態を stale として扱うこと

## 7. 非機能要件

### 7.1 性能要件

#### 要件ID: NFR-001

* 3,000〜10,000枚規模の写真を対象に運用可能とすること
* 大量件数でも処理が停止せず、途中経過を確認できること
* 3,000枚規模の一覧初期表示では、初回可視範囲のサムネイル表示を3秒以内とすること
* 通常のUI操作に対して500ms以内に画面応答があること

#### 要件ID: NFR-002

* サムネイル一覧表示は仮想スクロール等により大量件数でも実用上待ち時間が少ないこと
* 仮想スクロール等により大量件数を扱えること
* 基準環境において、3,000枚規模の技術評価全件処理を30分以内の目標とすること
* サムネイルおよびプレビュー配信はブラウザキャッシュを前提にし、`Cache-Control` と再検証可能なキャッシュ戦略を持つこと
* AI評価時間はモデル依存の参考値として別記し、単画像評価は1枚あたり5〜15秒、グループ内比較は1グループあたり10〜30秒程度を目安とすること
* AI評価の合否判定は疎通性と継続処理可否を優先し、技術評価ベースの暫定評価表示を阻害しないこと

### 7.2 可用性・復旧性

#### 要件ID: NFR-003

* 処理途中で中断しても再開しやすいこと
* 解析途中結果を段階保存すること
* AIサーバー疎通失敗時は解析開始前に検知し、利用者へ明確なエラーを提示すること

### 7.3 保守性

#### 要件ID: NFR-004

* AI推論プロバイダを差し替え可能な構造とすること
* モデル依存部分はAdapter層に分離すること

#### 要件ID: NFR-005

* 評価ロジックの重みや閾値は設定で変更可能とすること

### 7.4 リソース管理

#### 要件ID: NFR-006

* キャッシュディレクトリの保存先を設定可能とすること
* キャッシュ上限値を設定可能とすること
* キャッシュ再利用と古いキャッシュ削除方針を定義すること

### 7.5 セキュリティ

#### 要件ID: NFR-007

* 画像データは外部へ送信しないことを前提とする
* 通信先は localhost のみを基本とする
* 開発時のCORS許可オリジンは `localhost` 系に限定し、本番同一オリジン配信では追加CORS設定を不要とすること

### 7.6 ログ・監査性

#### 要件ID: NFR-008

* 解析ジョブの実行ログを保持すること
* AI応答、失敗理由、再試行状況を確認できること
* アプリケーションログは `var/logs/` 配下にローテーション付きで保持し、初期保持期間は30日とすること
* AIの生レスポンスや詳細リクエストは、DB要約保存と圧縮ファイル退避の併用を許容し、保持方針を定義すること

### 7.7 互換性

#### 要件ID: NFR-009

* OpenAI互換APIを利用すること
* LM Studio以外への切替余地を残すこと

## 8. 制約条件

### 8.1 技術制約

* ローカル実行を前提とする
* AIサーバーは別途起動済みであることを前提とし、解析開始前に疎通確認を行う
* AIモデルはOpenAI互換API越しに利用する
* 初版は単一ユーザー前提とする
* 初版のジョブ実行基盤は FastAPI アプリケーション内の in-process バックグラウンド実行を前提とする

### 8.2 運用制約

* ユーザーは処理対象フォルダへの読み書き権限を持つこと
* RAW現像品質の追い込みは本システムの責務外とする
* AIの評価は補助であり、最終採否はユーザーが決定する

## 10. システム構成設計

### 10.1 論理構成

1. フロントエンド
2. バックエンドAPI
3. ジョブ実行ワーカー
4. 画像処理モジュール
5. AI推論アダプタ
6. DB
7. キャッシュ領域
8. XMP書き戻しモジュール

### 10.2 処理フロー概要

1. ユーザーが対象フォルダを指定する
2. バックエンドがAIサーバー疎通と設定整合性を確認する
3. バックエンドが画像一覧を登録する
4. サムネイルと推論用プレビューを生成する
5. メタデータ抽出を行う
6. 類似度計算と時間近接性によりグルーピングする
7. 画像処理で技術評価を行う
8. 技術評価で候補を絞り込み、AIで意味的評価とグループ内順位付けを行う
9. AI未完了分には暫定評価を表示し、最終的に総合評価から★/rejectを決定する
10. UIへ結果を表示する
11. ユーザーが修正する
12. 必要に応じてXMPへ書き戻す

### 10.3 リポジトリ構成方針（固定）

本プロジェクトの実装リポジトリは、React フロントエンドと Python バックエンドを同居させるモノレポ構成で固定する。初版では以下のディレクトリ構成を標準とし、実装担当者は原則この構成に従うこと。

```text
/
├─ apps/
│  ├─ web/                  # React + Vite フロントエンド
│  │  ├─ public/
│  │  ├─ src/
│  │  │  ├─ app/
│  │  │  ├─ features/
│  │  │  ├─ components/
│  │  │  ├─ lib/
│  │  │  ├─ hooks/
│  │  │  ├─ types/
│  │  │  └─ routes/
│  │  ├─ package.json
│  │  ├─ vite.config.ts
│  │  └─ tsconfig.json
│  └─ api/                  # Python バックエンド（uv 管理）
│     ├─ src/
│     │  └─ skysort_api/
│     │     ├─ api/
│     │     ├─ services/
│     │     ├─ domain/
│     │     ├─ infra/
│     │     ├─ workers/
│     │     └─ main.py
│     ├─ tests/
│     ├─ pyproject.toml
│     └─ uv.lock
├─ packages/                # 任意の共通モジュール置き場
│  ├─ config/               # 共通設定、lint/format 設定ラッパ等
│  └─ client/               # OpenAPI 由来の TypeScript API client 等
├─ scripts/                 # 開発補助スクリプト
├─ docs/                    # 要件・設計・運用ドキュメント
├─ var/                     # ローカル実行時生成物（git 管理外）
│  ├─ cache/
│  ├─ data/
│  ├─ logs/
│  └─ tmp/
├─ .gitignore
├─ README.md
└─ package.json             # Node ワークスペース定義用
```

補足方針は以下の通りとする。

* `apps/web` は UI 実装専用とし、画像処理や AI 推論の本体ロジックは持たせないこと
* `apps/api` は API、バッチ、ジョブ実行、画像処理、AI 接続、XMP 書き戻しを包含するバックエンド実装の責務を持つこと
* Python の依存管理、仮想環境、実行は `apps/api/pyproject.toml` を基準に `uv` で統一すること
* Node 系のワークスペース管理は `pnpm` を標準とし、リポジトリルートの `package.json` と `pnpm-workspace.yaml` を起点に `apps/web` と `packages/*` を束ねること
* フロントエンドとバックエンドの契約は OpenAPI を正本とし、TypeScript 側で必要な API client や型定義は `packages/client` に生成または配置すること
* OpenAPI 由来の client 生成は API 契約変更時に明示的な生成コマンドで更新する運用を標準とすること
* 実行時に生成される DB、キャッシュ、ログ、一時ファイルは `var/` 配下へ集約し、Git 管理対象外とすること
* ユーザーの写真原本はリポジトリ配下へコピーしないこと。入力フォルダは外部パスを参照する前提とすること
* 将来デスクトップ化や配布形態変更を行う場合でも、`apps/web` と `apps/api` の責務境界は維持すること

### 10.4 バックエンド内部構成方針（固定）

`apps/api/src/skysort_api` 配下は、少なくとも以下の責務で分割する。

* `api/`: HTTP エンドポイント、リクエスト/レスポンス DTO、OpenAPI 公開
* `services/`: ユースケース調停、ジョブ起動、トランザクション境界
* `domain/`: 写真、グループ、評価、設定などのドメインモデルと判定ロジック
* `infra/`: DB、ファイルI/O、LM Studio 接続、EXIF/XMP、サムネイル生成など外部依存
* `workers/`: 長時間ジョブ、非同期処理、再解析処理

この分割により、API 層から画像処理や AI ベンダー依存実装へ直接アクセスしない構造を維持すること。

### 10.5 フロントエンド内部構成方針（固定）

`apps/web/src` 配下は、少なくとも以下の責務で分割する。

* `app/`: アプリ初期化、プロバイダ、グローバル設定
* `routes/`: 画面単位のルーティング定義
* `features/`: 取り込み、レビュー、設定など機能単位の UI と状態管理
* `components/`: 機能横断で再利用する表示部品
* `lib/`: API 呼び出し、ユーティリティ、日付変換など
* `hooks/`: UI 共通カスタムフック
* `types/`: UI 専用型、外部生成型の再エクスポート

画面固有のロジックは `routes/` へ肥大化させず、機能単位で `features/` に集約すること。

## 11. 詳細設計

本章は推奨設計であり、要件を満たす限り実装担当者が詳細実装を最適化してよい。ただし、本章で固定した安全方針、データ保持方針、API粒度は優先する。

### 11.1 グルーピング設計

グルーピングは以下を組み合わせて判定する。

* 撮影時刻差
* pHash等の簡易視覚類似度
* 画像埋め込みベクトルの類似度
* 必要に応じて構図特徴量

判定ルールは設定可能とする。初版では以下の流れを推奨する。

1. 撮影時刻順にソートし、取得可能な場合はサブ秒時刻を優先利用する
2. 一定秒数以内の画像を同一候補集合とする
3. 類似度閾値により同一バーストへ割り当てる
4. サブ秒時刻が取得できない場合はファイル順を補助情報として用いる
5. 不一致が大きい場合は別グループとする

### 11.2 技術評価設計

各画像に対し以下の指標を算出する。

* シャープネススコア
* モーションブレ推定値
* 明部飽和率
* 暗部潰れ率

技術評価総合点は設定ファイルで重み付けを行う。
シャープネスは画像全体の平均値ではなく、中央重点領域またはグリッド分割後の高エッジ領域を優先して算出する。
白飛び率と黒つぶれ率は設定可能とし、初期値はそれぞれ輝度 252/255 以上、3/255 以下の画素比率を用いる。
初版では被写体切れ推定および被写体占有率は必須対象外とし、将来拡張で検出モデル併用を検討する。
個別指標値は総合点とは別に永続化し、重み変更時の再集計に利用できるようにする。

### 11.3 AI意味評価設計

AIへは以下の二段階評価を行う。

#### 単画像評価

* 機体の見栄え
* 構図の自然さ
* 被写体の収まり
* ピントやブレの印象

#### グループ内相対評価

* グループ内で最も良い画像
* 似た画像との差分
* 削除候補

プロンプトはテンプレートファイルとして外部化し、写真メタデータ要約および必要な評価条件を埋め込んで送信する。
画像送信は OpenAI互換 `image_url` の data URL 形式による base64 インライン送信を標準とする。
単画像評価は全件必須ではなく、技術評価、グループサイズ、代表性をもとに候補へ優先実行する。技術評価で明らかな低品質と判定された画像は AI 評価をスキップできる。
複数画像比較では既定の推論用プレビュー長辺を1024pxとし、複数画像送信時は長辺512pxへの自動縮小を許容する。
候補を絞り込んだ上で、最大6枚ずつグループ内比較を行う。
グループが6枚を超える場合は候補集合を複数回に分けて比較し、比較結果を集約して最終順位を決定する。
AI応答は構造化JSONを正本とし、以下の順序で回復を試みる。

1. `response_format` 等によるJSONモード指定
2. 応答テキストからJSONブロックを抽出して再パース
3. 最大2回の再試行
4. なお失敗する場合は `ai_eval_failed` として記録し、暫定評価のままレビュー可能とする

AI送信内容、応答JSON、生レスポンス、テンプレート識別子、モデル名、応答時間は監査用に保持する。
複数画像比較の対象集合は、各応答に対して `target_photo_ids_json` 相当の形で保持し、どの画像群を比較した結果か追跡できるようにする。
AI応答JSONは schema version を持ち、後方互換が必要な場合でも解釈可能にする。

テンプレート雛形の例:

```text
[single_image_v1]
あなたは航空機写真の選別アシスタントである。説明文は不要で、指定JSONのみを返すこと。
評価対象:
- photo_id: {{ photo_id }}
- capture_time: {{ capture_time }}
- technical_score: {{ technical_score_total }}
観点:
- 機体の収まり
- 構図
- ブレやピントの印象
```

```text
[group_compare_v1]
あなたは同一バースト内の航空機写真を比較する。説明文は不要で、指定JSONのみを返すこと。
入力画像:
{{ candidate_photo_ids }}
観点:
- ベストカット
- 他画像との差分価値
- 削除候補
```

期待するJSON出力例:

```json
{
  "best_photo_id": "photo_001",
  "ranking": [
    {
      "photo_id": "photo_001",
      "rank": 1,
      "semantic_score": 91,
      "rarity_score": 84,
      "reason": "機体全体が収まり、背景との差分価値が最も高い"
    }
  ],
  "drop_candidates": ["photo_004"]
}
```

### 11.4 総合評価設計

以下のような重みを初期値とする。

* 技術品質: 35%
* 構図・見栄え: 35%
* 被写体状態: 20%
* グループ内希少性: 10%

グループ内希少性は、グループ内比較で得た「他カットとの差分価値」を 0〜100 に正規化したスコアとして扱う。
最終的な★評価変換は閾値ベースで行う。閾値は設定変更可能とする。
AI評価未完了時は技術評価ベースの暫定★評価を表示し、AI評価完了後に最終評価へ置き換える。
ユーザー補正は重み計算に含めず、算出後の手動上書きレイヤーとして扱う。

### 11.5 ユーザー補正設計

* 手動変更履歴を保存する
* 将来拡張として個人嗜好補正に利用できる構造とする
* 初版では学習機能は任意実装とし、少なくとも変更履歴保存までは必須とする
* 手動変更された `rating`、`selection_status`、`pick`、`best_cut` は、明示的にリセットしない限り自動再解析結果より優先する
* 再解析結果は提案値として保持できるが、ユーザー確定値を自動上書きしないこと

### 11.6 XMP書き戻し設計

XMP書き戻しは、標準互換を優先しつつ SkySort 独自情報を補助的に保持する。

| 内部状態 | 書き戻し先 | 方針 |
|---|---|---|
| `rating = 1..5` | `xmp:Rating` | 標準互換を優先して 1〜5 を書き込む |
| `selection_status = rejected` | `xmp:Rating = -1` | reject は標準互換の拒否表現を優先する |
| `pick = true` | `skysort:Pick` | SkySort 独自 namespace に真偽値で保持する |
| `best_cut = true` | `skysort:BestCut` | SkySort 独自 namespace に真偽値で保持する |
| `reviewed = true` | `skysort:Reviewed` | SkySort 独自 namespace に真偽値で保持する |

SkySort 独自 namespace は `skysort:` を使用し、他ソフトで可視化されない可能性があることを前提とする。
既存タグが存在する場合は評価関連タグのみを更新し、他の既存タグは維持する。

* 評価レコードはジョブごとに新規作成し、上書きではなく履歴を残す
* 現在表示する評価は、対象写真に対する「最新の有効な評価レコード」を取得して扱う
* `evaluation_status` は `provisional / final / ai_eval_failed / stale` を表現できるようにする
* `selection_status` は少なくとも `normal / rejected` を表現できるようにする
* `selection_status = rejected` の場合、`rating` は保持しない
* `best_cut_flag = true` は同一グループ内で1枚に限定する
* `composition_score`、`subject_state_score`、`rarity_score` は総合評価重み変更時の再集計に利用できるよう保持する

### 13.7 主テーブル: rating_history

想定カラム:

* id
* photo_id
* job_id
* old_rating
* new_rating
* old_selection_status
* new_selection_status
* changed_by_user
* changed_at
* reason

### 14.2 AIサーバー接続設計

* OpenAI互換エンドポイントを利用する
* モデル名、ベースURL、タイムアウト、AI同時実行数を設定可能とする
* 画像送信は OpenAI互換 `image_url` の data URL 形式による base64 インライン送信を標準とする
* AI応答が不正JSONの場合は、JSONモード指定、JSONブロック抽出、最大2回の再試行を順に行う
* 解析開始前に、モデル一覧取得に加えて指定モデルへの軽量な画像付き推論テストによるヘルスチェックを行う
* AI推論は逐次実行を初期値とし、画像処理前処理とは並列度を分離する
* 複数画像比較は最大6枚とし、対象モデルの制約で同時送信が困難な場合は段階比較で代替する
* `ai_eval_failed` となった画像またはグループは暫定評価のままレビュー対象に残す
* ヘルスチェックでは、少なくとも base URL 到達、指定モデル存在、指定モデルロード、vision入力可否、構造化JSON応答可否を確認する

## 15. ジョブ制御設計

### 15.1 バックグラウンド実行

* 解析は FastAPI アプリケーション内の in-process バックグラウンドジョブとして実行する
* 初版では pause/cancel をサポートせず、中断再開しやすい段階保存を優先する
* フロントエンドは API-003 による進捗取得を常時利用可能とし、SSE は任意の補助手段として利用する

### 15.2 ジョブ状態

* queued
* running
* completed
* failed

### 15.3 重複排除とキャッシュ戦略

* 同一ファイルに対する再取込は重複登録しないこと
* 解析済み結果はキャッシュ利用を優先すること
* 更新済みファイルはdirty判定により再解析対象へ回すこと
* サムネイルとプレビューは `var/cache/` に保持し、HTTP配信時は `Cache-Control` と再検証可能なヘッダを付与すること
* SkySort 自身による評価関連XMP更新のみが行われた場合、その変更は画像再解析のdirty判定対象外とすること
* Phase 2 のグループ結合・分割後の評価は自動再解析せず、stale化した上で明示操作により再解析すること

### 15.4 起動排他とDB保護

* 既定ポートが使用中の場合はバックエンド起動を失敗させ、二重起動を避ける
* SQLite は WAL モードを前提とし、単一アプリケーションからの継続利用を基本とする
* SQLite の busy timeout を設定し、長時間トランザクションを避けること
* DBファイルまたはロックファイルに対する排他制御を行い、同一データ領域への多重書き込みを避ける

## 17. エラー処理設計

### 17.1 想定エラー

* 画像読み込み失敗
* RAW変換失敗
* EXIF抽出失敗
* AIサーバー接続失敗
* AIサーバーヘルスチェック失敗
* AI応答JSON不正
* AI評価失敗終端（`ai_eval_failed`）
* XMP書き戻し失敗
* XMP競合更新失敗
* ExifTool 未導入または実行失敗

### 17.2 エラー時方針

* 単一画像失敗で全体処理を停止しないこと
* エラー内容をジョブログへ残すこと
* AI応答JSON不正時は JSONモード指定、JSON抽出、最大2回再試行を行うこと
* 再実行可能なものはリトライすること
* 最終的に失敗一覧として確認できること
* AIサーバー疎通失敗時は解析開始自体をブロックすること
* `ai_eval_failed` の対象は暫定評価のままレビュー可能とすること
* ExifTool 未導入時は書き戻し機能のみを失敗させ、レビューや評価閲覧は継続可能とすること
* XMP競合時は上書きせず、dry_run結果で利用者へ差分を提示すること
* 手動変更された `rating`、`selection_status`、`pick`、`best_cut` は自動再解析で上書きしないこと
* SkySort 自身による評価関連XMP更新のみが検出された場合は dirty とみなさないこと

## 18. 受入条件

### 18.1 必須受入条件

1. ローカルフォルダから3,000枚以上の画像を取り込めること
2. 類似写真をグルーピングして表示できること
3. 各画像に技術評価スコアが付与されること
4. AI評価により候補画像からグループ内ベスト候補が、二段階比較方式で選出されること
5. AI評価未完了時でも各画像に暫定★評価を表示でき、AI完了後に★1〜★5または reject の最終評価へ更新できること
6. UIで結果確認と手動修正ができること
7. 評価結果を保存できること
8. ARWはサイドカーXMP、JPEGは埋め込みXMPとして安全に書き戻せること
9. AI接続先URLとモデル名を設定変更できること
10. 処理途中で中断しても再開しやすいこと
11. XMP書き戻しは `dry_run` を既定とし、差分確認後に実行できること
12. AIサーバー未起動時に事前チェックで検知し、解析開始前にエラー表示できること
13. 各グループのベストカットが1枚に保たれ、複数高評価と混同されないこと
14. 手動補正が再解析で自動上書きされないこと

## 24. 想定課題と留意事項

* RAW現像条件により見た目が変化するため、推論用プレビュー生成条件は固定化する必要がある
* AIの評価は毎回完全一致しない可能性があるため、必要に応じてキャッシュ利用と再評価UIを設ける
* 「かっこいい」等の主観評価は個人差があるため、手動補正を前提とする
* 解析速度はモデルサイズ、量子化方式、解像度、同時実行数に依存するため、初期チューニングが必要である
* AI評価は3,000枚規模で半日以上を要する可能性があるため、候補絞り込みと暫定評価表示を前提とする
* 複数画像同時送信の上限はモデル依存で変動し得るため、段階比較へのフォールバックを維持する
* PNGは初版でXMP書き戻し対象外とし、入力表示用途に限定する
