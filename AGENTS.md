## 2. プロジェクトの目的

`SkySort` は、`skysort` リポジトリで開発する航空機写真向けのローカル完結型 AI 写真選別ソフトです。大量の連写写真を対象に、以下を行う前提で設計します。

- 写真の取り込み
- サムネイルと推論用プレビュー生成
- メタデータ抽出
- 類似写真グルーピング
- 技術評価
- AI による意味評価
- 最終レーティング付与
- UI 上での確認・修正
- XMP Rating 等への書き戻し

重要な前提:

- 完全ローカル動作を前提とし、remote AI 接続は明示 opt-in の例外として扱う
- AI 接続先は LM Studio の OpenAI 互換 API を第一候補とする
- 将来の推論プロバイダ差し替えを前提に、外部依存は疎結合に保つ
- AI 単独ではなく、画像処理の定量指標と組み合わせて評価する
- 絶対評価より、連写グループ内の相対比較を重視する
- 最終判断は必ずユーザーが上書きできるようにする

## 3. 現在のリポジトリ状態

## 4. 固定方針

### 4.1 アーキテクチャ

- フロントエンドは `apps/web` に閉じ込める
- バックエンドは `apps/api` に閉じ込める
- UI から画像処理や AI 推論実装へ直接依存させない
- フロントエンドとバックエンドの契約は OpenAPI を正本とする
- ランタイム生成物は `var/` 配下へ集約する
- ユーザーの写真原本をリポジトリ配下へコピーしない

### 4.2 バックエンド責務分離

`apps/api/src/skysort_api` では、少なくとも以下の分割を維持してください。

- `api/`: HTTP エンドポイント、DTO、OpenAPI
- `services/`: ユースケース調停、ジョブ起動、トランザクション境界
- `domain/`: ドメインモデル、判定ロジック
- `infra/`: DB、ファイル I/O、AI 接続、Exif/XMP、画像処理など外部依存
- `workers/`: 長時間ジョブ、非同期処理、再解析

`api/` から `infra/` の詳細実装へ直接飛ばず、`services/` または `domain/` を介してください。

### 4.3 フロントエンド責務分離

`apps/web/src` では、少なくとも以下の分割を維持してください。

- `app/`: 初期化、プロバイダ、グローバル設定
- `routes/`: 画面単位のルーティング
- `features/`: 取り込み、レビュー、設定などの機能単位実装
- `components/`: 機能横断の共通表示部品
- `lib/`: API 呼び出し、ユーティリティ
- `hooks/`: 共通カスタムフック
- `types/`: UI 専用型、生成型の再エクスポート

画面固有ロジックを `routes/` に肥大化させないでください。

## 5. 実装原則

- 要件にない近道のために、ローカル完結前提を崩さない
- AI プロバイダ固有コードをドメインや API 層へ漏らさない
- 解析ジョブは中断・再開・再実行を考慮して設計する
- UI が過去作業を失わないよう、ユーザー向けの永続単位はサーバー保存の `projects`、解析実行履歴は `jobs` として扱う
- cancel は graceful cancel とし、途中結果を保存して `canceled` にする。同一 `job_id` の途中 resume ではなく、新規 job 作成とキャッシュ再利用で復旧する
- キャッシュ、途中結果、監査ログを前提にする
- グループ結合・分割後は、関連評価を自動で確定させず `stale` として扱えるようにする
- AI 応答は構造化 JSON を前提にし、自然文だけに依存しない
- 手動補正は再計算値とは別レイヤーで保持する

## 6. データと安全性

- 外部ネットワークへの画像送信を前提にしない。remote AI はユーザーが明示的に許可した場合のみ扱う
- 通信先は原則 `localhost` に限定し、remote AI を許可する場合も設定と監査上で明確に区別する
- ARW へ直接書き込まない。XMP サイドカー更新を基本とする
- 既存メタデータは保守的に扱い、評価関連タグのみ更新する
- ユーザーの原本・既存タグ・手動評価履歴を壊す変更は避ける

破壊的な変更を伴う処理では、必ずドライランまたは事前確認可能な設計を優先してください。

## 7. 実装時の優先順位

機能追加や設計判断では、次の優先順位で整合性を取ってください。

1. ローカル完結・原本保護・再現性
2. `docs/plan.md` の固定方針への適合
3. 疎結合な責務分離
4. 大量件数でも破綻しない性能と段階保存
5. UI の操作性と手動補正しやすさ

## 8. テストと検証

実装時は、変更範囲に応じて必要な検証を追加してください。

- バックエンド: API 契約、ドメイン判定、ジョブ進捗、再解析、XMP 書き戻しの安全性
- フロントエンド: 主要画面の状態遷移、一覧表示、レビュー操作、ショートカット
- AI 連携: JSON スキーマ逸脱時の再試行、ヘルスチェック、タイムアウト、監査記録

テスト方針:

- 単体テストを書ける責務分割を優先する
- 重い E2E だけに依存しない
- AI 実サービス呼び出し前提の不安定なテストは常設しない
- モデル応答はモック可能な境界を用意する

## 9. コーディングルール

- 小さく分割し、責務を越えるファイル肥大化を避ける
- 設定値、閾値、重みはコードへ直書きせず設定可能にする
- プロンプトはテンプレートファイル化し、名前またはバージョンを識別可能にする
- ログ、DB 保存値、API 返却値で命名や状態表現を揃える
- 将来の Windows 11 主対象を意識し、パス処理や外部コマンド呼び出しを雑に実装しない

## 10. 作業の進め方

- 変更前に、既存の要件・設計・実装を確認してから編集する
- 関連ファイルだけで完結するか、責務境界を跨ぐかを先に判断する
- 新しい依存や新しいトップレベル構成を足す場合は、`docs/plan.md` と整合する理由を説明できる状態にする
- 要件に影響する変更では、必要なら `README.md` や `docs/` も更新する

## 11. このリポジトリで避けること

- UI 側へ画像処理や AI ベンダー依存ロジックを埋め込むこと
- 単一巨大モジュールへ処理を寄せること
- 原本ファイルの取り扱いを雑にすること
- OpenAPI を無視してフロントエンドとバックエンドを直接ねじ込むこと
- 一時ファイル、DB、キャッシュ、ログを散在させること
- 要件で定義された `reject` と ★1 を同一概念として潰すこと

## 12. AGENTS.md の運用ルール

このファイルは短く保ってください。詳細仕様やテーブル定義、画面項目、API 詳細は `docs/plan.md` へ寄せます。

`AGENTS.md` に追記すべき内容は次に限定します。

- エージェントが実装判断で頻繁に使う固定ルール
- 間違えると設計破綻しやすい責務境界
- 安全性やデータ保護に直結する注意点
- 日常的に使う開発コマンドや検証フロー

現在の日常コマンド:

- バックエンド起動: `pnpm dev:api`
- バックエンド migration のみ: `pnpm migrate:api`
- フロントエンド起動: `pnpm dev:web`
- バックエンド構文確認: `uv run python -m compileall -q apps/api/src`
- バックエンドテスト: `uv run --project apps/api pytest`
- OpenAPI/client 更新: `pnpm generate:client`
- フロントエンドテスト: `pnpm --filter @skysort/web test`
- フロントエンド型確認: `pnpm --filter @skysort/web typecheck`

設計変更、仕様変更、実装変更によってこのファイルの内容が古くなった場合は、関連変更と同じタイミングで直ちに更新してください。後続タスクでまとめて直す前提にしないでください。

長い背景説明や重複仕様は追加しないでください。
